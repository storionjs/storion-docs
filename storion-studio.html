Storion Studio

Storion Studio is a Chrome extension that turns your browser's localStorage into a structured database you can manage with a visual UI—like pgAdmin for the browser. It uses the same data layout as the @storion/storion npm package, so you can use the extension as an admin console while your webapp uses the library.

Features

  • Create, delete, and organize multiple databases
  • Create tables with custom columns and manage schema
  • Insert, read, update, and delete rows with a table grid
  • Query panel: filter and sort with the same JSON query language as Storion
  • Export/import databases as JSON
  • Real-time storage usage (5MB localStorage limit)
  • Optional: stream change events to your webapp via postMessage or Chrome messaging

Installation

  1. Clone or download the Storion Studio extension folder
  2. Open Chrome and go to chrome://extensions/
  3. Enable Developer mode (top-right)
  4. Click Load unpacked
  5. Select the Storion Studio folder
  6. The Storion Studio icon appears in your toolbar

Quick start

  1. Click the Storion Studio icon
  2. Click + New next to "Databases", name it (e.g. myAppDB), click Create
  3. Select the database, click + New next to "Tables", add a table (e.g. users) with columns
  4. Click + Add Row to insert data; use the edit/delete icons on rows to update or remove
  5. Use the Query panel to filter and sort with the same JSON query format as db.query()

Data storage

All data is stored in a single localStorage key: __LS_DB__. The structure matches what the Storion library uses, so databases you create in the extension are compatible with createDatabase({ name: '...', storage: 'localStorage' }) in your app.

Using Storion Studio with your webapp

Storion Studio models the same data layout as the @storion/storion package. You can use the extension as a separate "admin console" and stream change events to your webapp.

At the library level, Storion provides:

  • db.setChangeBroadcaster(broadcaster) – after each insert/update/delete/createTable/deleteTable, you can forward a normalized StorionChangeEvent to another context (e.g. via chrome.tabs.sendMessage, postMessage, or BroadcastChannel).
  • createChangeListener(transport, onChange) – on the receiving side, subscribes to a custom transport and calls onChange(event) when a valid change event arrives.

Example with window.postMessage (consumer in your webapp):

import { createChangeListener } from '@storion/storion';

const transport = {
  onMessage(handler) {
    function listener(ev) {
      if (!ev.data || ev.data.source !== 'storion-change') return;
      handler(ev.data.payload);
    }
    window.addEventListener('message', listener);
    return () => window.removeEventListener('message', listener);
  }
};

const stop = createChangeListener(transport, (event) => {
  console.log('Change from Storion Studio:', event);
  // e.g. refetch data or update local UI state
});

This wiring is optional: Storion Studio works offline by default; the extension can broadcast changes when you want to sync with a page that uses the Storion library.

Limitations

  • 5MB storage limit – localStorage has a ~5MB limit per origin; the extension shows usage.
  • Single origin – Data is isolated per origin (protocol + domain + port).

Query language in Studio

The Query panel in Storion Studio uses the same JSON query format as db.query(): where, orderBy, limit, offset. See Query language for full syntax and operators.
